UI Design & Design Systems

The system behind the screen.

Research informs what to build. Design systems determine whether it can be built consistently, at scale, and maintained by people who weren't in the room when decisions were made. This page shows the UI craft and system thinking behind two of the engagements in this portfolio — Compartamos Banco and Whisper BI.

Exhibit 01

Design tokens & brand system

Compartamos Banco · Mobile app & Contact Center

The Compartamos DLS is built on a single brand constraint: the magenta (#E8007D) must be instantly visible in direct sunlight on a low-end Android screen. Everything else — typography scale, spacing, surface colors, state indicators — derives from that constraint.

Design decision The color system uses the brand magenta exclusively for primary actions and critical status indicators. Secondary interactions use a teal accent — distinct enough to carry information independently from the primary color, compliant with WCAG AA contrast ratios on both light and dark surfaces.

Exhibit 01B

Spacing scale & grid system

Compartamos Banco · Mobile 4-col · Web 12-col

The spacing system is built on an 8pt base grid — every space value is a multiple of 8, with a 4px exception for fine adjustments. This constraint kept the mobile app visually consistent across 12+ screens built by different designers and contractors over 3 years without a design review bottleneck.

Why two different grids The mobile app uses a 4-column grid (16dp margin, 8dp gutter) optimized for 360dp viewports — the target device was an entry-level Android at 5". The Contact Center web tool uses a 12-column grid (40px margin, 24px gutter) anchored to 1280px — the target was a 13" laptop in an office environment. Same design system, different spatial logic for each context.

8pt base grid · Mobile 4-col (360dp) · Web 12-col (1280px) · 48dp minimum touch target

Exhibit 02

Component library

Compartamos Banco · Built in Figma

The component library covers the full interaction surface of the field agent app and Contact Center tool. Every component is documented with usage rules, state variations, and accessibility annotations — so designers joining the project mid-stream and external contractors could work without a design review on every screen.

Design decision Input components use floating labels rather than static labels above fields — a deliberate choice for a small-screen context where vertical space is at a premium. The label collapses into the field on focus, recovering a full field height worth of vertical space per input.

Exhibit 02B

Atoms — icons, badges, chips, progress

Compartamos Banco · Full atom library

Atoms are the smallest reusable units in the system — they don't compose other components, they are composed. Every icon, badge, chip, and progress indicator here has a defined usage rule, a documented state set, and a specific meaning in the Compartamos product language.

Status badges as a design language The six pipeline states — Perfilamiento, Validación, Llamada cortada, En atraso, Completado, Nuevo — are not just UI labels. They are operational definitions that determine what action a Contact Center agent takes next. The color semantics (green = active qualification, amber = follow-up needed, red = past due) were validated with the operations team before implementation. A design decision that looked cosmetic was actually a process definition.

Icons (24×24dp · 3 states) · Status badges (6 pipeline states) · Chips (4 variants) · Progress (linear + step) · Avatar system (4 sizes + stack)

Exhibit 02C

Molecules — form groups & list items

Compartamos Banco · All states documented

Molecules are where atoms become functional. The form group molecule — label + field + helper text + states — is the primary data-collection pattern in the Compartamos product. It appears 47 times across the loan origination flow. Getting it right once meant every instance was right.

The floating label decision Floating labels — which collapse from placeholder position to a small label above the field on focus — were chosen over static labels for one specific reason: vertical space on a 5" Android screen. Each floating label recovers the equivalent of one field height per input. In a form with 12 fields, that compounds into a measurable reduction in scroll depth, which directly reduced the abandonment rate at the data-capture step.

Exhibit 02D

Handoff annotation — engineering specs

Compartamos Banco · CC-004 Registro Duplicado

A design system is only as good as the handoff that bridges it to engineering. This annotation document for the Registro Duplicado screen — one of the most complex in the Contact Center tool — shows how design decisions were communicated: layout specs, typography tokens, color references, component names, and interaction behavior in a single document.

The handoff format Rather than relying on Figma comments alone (which engineers rarely read in sequence), I produced structured spec sheets for complex screens — one page per screen, organized by concern: layout first, then typography, then color tokens, then components referenced, then interaction behavior. This reduced the back-and-forth between design and engineering by making every decision explicit before implementation started.

Exhibit 03

Mobile app — field agent interface

Compartamos Banco · Android · 15,000+ field agents

The field agent app had to work on entry-level Android devices, under variable lighting, with intermittent connectivity. UI decisions were made with those three constraints as hard limits — not aspirations.

Key UI constraints Minimum touch target: 48×48dp. No gesture-only interactions — every action needed a visible tap target. Offline-capable state management: the sync status indicator (green dot + "Sincronización completa") was designed to be the first thing a user sees — before any data. If you don't know whether your data is current, you can't trust what you're looking at.

Home · Menú · Mi Menú — the three entry points of the field agent workflow. Role-based: promotoras see a 3-item home grid; asesores CI see an expanded menu with additional product lines.

Exhibit 04

Contact Center — web tool

Compartamos Banco · Web · Responsive

The Contact Center tool served a different user than the mobile app — office-based agents handling inbound calls, managing large data volumes, resolving exceptions. The design shifted from the mobile-first, thumb-friendly patterns of the field app to a denser, information-rich layout optimized for a 13"+ monitor.

The most complex screen The duplicate record resolution screen — shown here — was the hardest UX problem in the CC tool. When an incoming caller's data matches two existing records with different names or dates, the agent must decide which record is correct and archive the other. The side-by-side comparison layout was tested with agents before finalizing: initial versions used a sequential reveal (show one, then the other) — agents found they couldn't compare without scrolling. Side-by-side eliminated that entirely.

Registro duplicado — side-by-side comparison layout. Folio, age, birthdate, birthplace, and contact visible for both records simultaneously. The non-selected record auto-archives on confirmation — no supervisor escalation required.

Exhibit 05

Whisper BI — conversational analytics

B2B SaaS · Dark theme · AI-powered

The visual language of Whisper BI had to signal "enterprise intelligence" without the cold sterility of traditional BI tools. Dark background, electric blue primary actions, high-contrast data visualization — legible at a glance, familiar to a business user who has seen Slack and Notion.

AI-assisted design workflow This interface was produced using an AI-assisted design workflow — Claude integrated with Figma — which reduced iteration time by approximately 60%. The workflow is worth noting not as a novelty, but as a skill: knowing when to use AI generation vs. manual craft, and how to quality-control the output, is now a core competency in the design stack.
Design system contrast: mobile vs. web vs. AI-native

These two products — Compartamos and Whisper BI — represent opposite ends of the design constraint spectrum. Compartamos: light theme, maximum contrast, large touch targets, offline-capable, for a user who may be using the phone with one hand in the sun. Whisper BI: dark theme, dense information, keyboard-navigable, for a user at a desk who needs to see data and act on it in the same view. The underlying system thinking is identical — the visual output is intentionally different because the context of use is different.
